2 当前问题诊断

问题 1 — 消融设计缺陷：ERFM only 配置无拥堵事件注入，而其他配置有。导致 ERFM only 的 Cost 最低（197K vs 253-268K），不具备可比性。所有配置必须面对相同的拥堵场景。
问题 2 — AOCC 计算失真：3/4 配置的 AOCC 完全相同（0.9472），说明 L/U 边界计算有问题或所有实验的收敛轨迹几乎无差别。
问题 3 — 边映射覆盖率不足：仅 764/1769 (43.2%) 的 SUMO 边通过 OSM 节点 ID 匹配成功。坐标体系差异（SUMO 本地投影 vs LE-DEGN UTM）导致空间匹配失败，而纯拓扑匹配受 netconvert 简化影响。
问题 4 — SA-DGWN 未正式集成：训练模型使用了临时的 reg_head，与标准 SADGWN 的 out_proj+Sigmoid 不一致。消融脚本中手动计算拥堵，未通过 LEDEGNSystem.predict_congestion()。
问题 5 — 缺少制度化验证：未做多 seed 重复实验，结果受随机性影响大。无可视化报告，无结构化实验结果文件。

3 最优改进计划（5 个阶段）

阶段 A — 提升边映射覆盖率 (目标 > 70%)

策略：OSM 节点 ID 拓扑匹配 + 坐标归一化回退匹配

  1. 将 SUMO 坐标从本地投影反算回 WGS84 (lat/lon)，再投影到 UTM 与 LE-DEGN 对齐
  2. 先做拓扑匹配（已有 43.2%），剩余未匹配边做坐标最近邻（阈值 30m）
  3. 重建 traffic_tensor.npy 和 edge_mapping.json
  4. 验证：匹配率 > 70%，时间方差 > 0.001

阶段 B — SA-DGWN 模型规范化集成

策略：创建 SUMOCongestionPredictor 包装类，适配 SADGWN 的标准接口

  1. 修改 sadgwn.py 增加 predict_speed_regression() 方法和 load_sumo_weights()
  2. 在 LEDEGNSystem.predict_congestion() 中集成 SUMO 数据源
  3. 验证：predict_congestion() 输出 σ_max > 0.7 且 >0.5 的边 ≥ 5%

阶段 C — 重新设计消融协议

策略：统一拥堵场景，仅改变应对机制

配置拥堵事件拥堵预测重规划逻辑
A (ERFM only)3 个 _default_events_real无ERFM 采样重规划（无预测能力）
B (ERFM+SA-DGWN)3 个 _default_events_realSUMO 模型预测拥堵边 → 提前规避
C (ERFM+LHH)3 个 _default_events_real无LHH 语言启发式逃逸
D (Full)3 个 _default_events_realSUMO 模型SA-DGWN 预测 + LHH 逃逸

关键保证：四组配置使用完全相同的初始贪心 NN 路径、相同的 3 个拥堵事件、相同的 30s time_budget。

阶段 D — 多 seed 制度化验证

策略：5 个随机 seed × 4 配置 = 20 次实验

  1. seeds = [42, 123, 456, 789, 1024]
  2. 每 seed 固定初始路径和拥堵事件
  3. 收集：final_cost, AOCC, replan_count, escape_count, congestion_detection_rate
  4. 输出 mean ± std 表格和箱线图

阶段 E — 生成最终报告并发布

策略：HTML 报告 + GitHub PR 合并

  1. 使用 html-report 技能生成含图表的完整实验报告
  2. 报告内容：方法回顾 → 实验设计 → 消融结果 → 组件贡献分析 → 结论
  3. Git 提交所有最终产出 → 推送 → 合并 PR #1
